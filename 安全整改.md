# WeClaw 安全整改文档

> 整理 WeClaw 部署环境的安全隐患分析和加固措施，按优先级排列

---

## 一、威胁模型分析

### 当前部署架构

```
外网
 │
 ▼
服务器 80 端口（Nginx）
 │
 ▼  /wechat 路由
Node.js 3000 端口（WeClaw）
 │
 ▼
本地文件系统（config/data/memos.md 等）
```

### 主要威胁来源

| 威胁 | 描述 | 风险等级 |
|------|------|---------|
| **伪造微信请求** | 攻击者构造微信格式的 POST 请求，触发 AI 执行操作 | 中（有签名验证） |
| **暴力扫描/探测** | 机器人扫描 80 端口，探测接口信息 | 低（无敏感路由） |
| **服务器入侵** | 攻击者拿到 shell，读取本地文件和密钥 | 高（一旦入侵后果严重） |
| **DDoS/耗尽资源** | 大量请求打垮服务或耗尽 AI Token 配额 | 中 |
| **Token/密钥泄露** | `.env` 文件、日志中出现明文密钥 | 高 |

---

## 二、已有防护（勿重复实现）

### 1. 微信签名验证 ✅

WeClaw 在 `src/wechat/xml.ts` 中对每个请求验证签名：

```
signature = SHA1(sort([token, timestamp, nonce]).join(''))
```

**只要 `WECHAT_TOKEN` 不泄露，攻击者无法伪造合法请求**，即使访问到 80 端口也无法触发 AI 执行操作。

### 2. 配置加密存储 ✅

用户通过 `/set` 命令设置的 API Key 使用 AES-256-GCM 加密存储，不以明文出现在日志或 AI 上下文中。

### 3. 备忘录文件不对外暴露 ✅

`config/data/memos.md` 是服务器本地文件，WeClaw 没有提供静态文件服务路由，**外网无法通过 HTTP 直接访问此文件**。

## 三、待加固事项

### 🔴 P0：高优先级

#### 1. 确保 WECHAT_TOKEN 足够复杂

**风险**：Token 过于简单（如 `hello123`）可能被暴力破解，导致请求伪造。

**措施**：

```bash
# 生成强随机 Token（32字节十六进制，64位）
openssl rand -hex 32
```

然后更新 `.env`：

```bash
WECHAT_TOKEN=生成的随机字符串
```

同步更新微信公众号后台的 Token 配置。

---

#### 2. 服务器 SSH 安全加固

**风险**：使用密码登录或默认端口，容易被暴力破解。

**措施**：

```bash
# /etc/ssh/sshd_config
PermitRootLogin no           # 禁止 root 直接登录
PasswordAuthentication no    # 禁止密码登录，只允许密钥
Port 22222                   # 修改默认端口（可选）
```

```bash
# 重启 SSH 服务
systemctl restart sshd
```

**确保先把你的公钥加入 `~/.ssh/authorized_keys`，再禁用密码登录，否则会锁死！**

---

### 🟡 P1：中优先级

#### 3. Nginx 限速防暴力扫描

**风险**：无限速时，攻击者可以高频请求耗尽 AI Token 配额或打垮服务。

**措施**：在 Nginx 配置中添加限速：

```nginx
# /etc/nginx/nginx.conf 的 http 块中添加
limit_req_zone $binary_remote_addr zone=wechat_limit:10m rate=10r/s;
```

```nginx
# /etc/nginx/sites-available/weclaw（你的站点配置）
server {
    listen 80;
    server_name your-domain.com;

    location /wechat {
        # 限速：每个 IP 每秒最多 10 个请求，突发最多 20 个
        limit_req zone=wechat_limit burst=20 nodelay;

        # 只允许 GET（验证）和 POST（消息推送）
        limit_except GET POST {
            deny all;
        }

        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # 拒绝访问其他路径（可选，根据需要）
    location / {
        return 404;
    }
}
```

```bash
# 验证配置并重载
nginx -t && nginx -s reload
```

---

#### 4. 服务器防火墙基础配置

**风险**：不必要的端口暴露在外网，增加攻击面。

**措施（以 ufw 为例）**：

```bash
# 重置规则
ufw reset

# 默认拒绝所有入站
ufw default deny incoming

# 允许 SSH（使用你修改后的端口）
ufw allow 22222/tcp   # 改成你的 SSH 端口

# 允许 HTTP（微信 Webhook）
ufw allow 80/tcp

# 允许 HTTPS（如果有证书）
ufw allow 443/tcp

# 启用防火墙
ufw enable

# 查看规则
ufw status verbose
```

**注意**：3000 端口（Node.js）只需要本机访问，不要对外开放，Nginx 会代理。

---

#### 5. 关于微信服务器 IP 白名单

**分析**：

- 微信服务器 IP **不固定**，官方虽有 IP 查询接口，但 IP 段会不定期变化
- IP 段范围很大（几十个 CIDR），且需要持续维护更新
- **维护成本 > 收益**，不推荐作为主要防护手段

**如果仍想尝试**，可查询微信当前 IP 段：

```bash
# 获取微信服务器 IP 列表（需要有效的 access_token）
curl "https://api.weixin.qq.com/cgi-bin/getcallbackip?access_token=YOUR_ACCESS_TOKEN"
```

通过 Nginx 日志分析历史来源 IP（仅供参考）：

```bash
# 分析访问 /wechat 的 IP 来源
grep "POST /wechat" /var/log/nginx/access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head -20
```

### 🟢 P2：低优先级

#### 6. 启用 HTTPS

**风险**：HTTP 明文传输，微信消息内容在传输过程中可被中间人截获。

**措施**：使用 Let's Encrypt 免费证书：

```bash
# 安装 certbot
apt install certbot python3-certbot-nginx

# 申请证书（需要有域名且 DNS 已解析到服务器）
certbot --nginx -d your-domain.com

# 自动续期
certbot renew --dry-run
```

---

#### 7. 日志中敏感信息检查

**风险**：调试日志可能打印出 API Key、用户消息等敏感信息。

**措施**：检查 Nginx 和 Node.js 日志：

```bash
# 检查 Nginx access log 中是否有敏感参数
cat /var/log/nginx/access.log | grep -i "key\|token\|secret\|password"

# 检查 Node.js 日志
cat logs/app.log | grep -i "key\|token\|secret"
```

确保 `.env` 文件权限正确：

```bash
chmod 600 .env
chmod 600 config/data/*.enc.json
```
